# Xenopairings — Design Document

## §1 Vision

Xenopairings is a lightweight Warhammer 40K tournament manager. Organizers create tournaments and get a manage URL by email. Players sign up with name, email (optional), army/faction, and army list. The organizer sets how many rounds and can auto-pair using Swiss rules. After each round, scores are entered and standings update live. No user accounts; everything works via token-bearing URLs.

## §2 Roles

| Role       | Identity carrier      | Capabilities |
|------------|-----------------------|--------------|
| Organizer  | Manage token in URL   | Create tournament, manage players, create rounds, trigger pairing, mark complete |
| Player     | Edit token in URL/cookie | Register, edit registration, withdraw |
| Viewer     | No token (anonymous)  | View tournament info, standings, pairings |

## §3 Scope

- Tournaments with configurable rounds and max players
- Swiss-style pairing: random round 1, then by wins / total points, no repeat matchups
- Per-round mission layout text
- Match scoring (battle points for each player)
- Standings: sorted by wins desc, then total points desc
- Army list text storage per player

Out of scope (v1):
- User accounts / login
- Multiple organizers per tournament
- ELO / rating systems
- Tiebreaker beyond total points

## §4 Tech stack

- ASP.NET Core Blazor Server (.NET 10)
- EF Core 10 + SQLite
- Hangfire (background jobs — nightly backup)
- Docker (Alpine image, non-root appuser)
- Bootstrap 5 + custom CSS (dark/light mode)
- Resend API for transactional email

## §5 Data model

### Tournament
Core entity. Slug identifies it in URLs. ManageToken is the organizer's credential.

### Player
Belongs to a Tournament. EditToken is the player's credential. IsDropped marks withdrawal.

### Round
Belongs to a Tournament. RoundNumber is 1-indexed. MissionLayout is free text.

### Match
Belongs to a Round. Player1 and Player2 (nullable for bye). Player1Score / Player2Score are nullable until entered. IsScored = true when both scores entered.

## §6 Pages

| Route | Page | Access |
|-------|------|--------|
| `/` | Index | Public |
| `/new` | NewTournament | Public |
| `/created/{id:guid}` | TournamentCreated | Public (shown once after creation) |
| `/t/{slug}` | TournamentPage | Public |
| `/t/{slug}/manage?t={token}` | ManageTournament | Organizer only |
| `/t/{slug}/edit-registration?t={editToken}` | EditRegistration | Player only |

## §8 Identity model

Identical pattern to consid_maltid:
- No user accounts
- Organizer identity = random 22-char URL-safe token in manage URL
- Player identity = random 22-char URL-safe token, delivered by email or stored in cookie
- Tokens generated by `TokenGenerator.RandomUrlSafeString(22)`
- Email addresses stored lowercase in DB
